As a software engineer, I have spent my career thinking about the design of stable systems, and one of the most fundamental features of a stable system is reproducibility across environments. Reproducibility is also a core value of experimental research. Reproducibility creates trust. Trust in a product or platform. Trust in experimental results.

While Jupyter Notebooks are incredibly powerful for exploratory analysis or ad hoc research, I find the use of them for deeper research to be a bit nerve-racking. Often, model analysis code is scattered and tangled throughout a notebook. A model is defined and trained across a few cells. Computations are run against training data in other cells. Visualizations are crafted across others. Any change I make risks breaking every cell in the notebook, or worse - it could lead to inaccurate interpretations or conclusions.

In my research, I want to be able to study models as they evolve over training, not their final output. In practice, this means that I need to be able to analyze variations of a single model architecture across multiple checkpoints captured during training. For the Modulo Addition model, I'm generally analyzing between 94 to 150 checkpoints. In order to compare analysis across checkpoints and across variations, I would need my data and visualizations to be as consistent as possible, which is simply unmanageable in a notebook.

To solve this problem, I needed to be able to lock down three things: training, analysis, and visualization. My approach for supporting re-use in training and analysis was to generate static outputs on disk wherever possible in a run-once, re-use repeatably paradigm. During training, checkpoints and training metadata are stored in safetensor files and json files respectively. During analysis, tensors, arrays, and summary data are exported to npz files and json files. From a reproducibility standpoint, this separation allows me to add new analysis functions that are guaranteed to run against the same underlying checkpoint data. I can also create or tweak visualizations without worrying about consistency in the underlying analysis data.

For visualizations, I've adopted Plotly for the core visualization library so that all visualizations are consistent in look, feel, and interactivity. By leveraging Plotly's standard figure objects, I am able to re-use the same views in notebooks, in web front-ends, or exported to disk. Changes made to individual visualizations have no impact on the underlying data.

One major unintended benefit to this approach is that I don't have to be stingy with analysis - at least not with small toy models. Analysis functions that might take a long time to compute, such as PCA analysis across all checkpoints and weight matrices, runs once. All subsequent loads of that PCA data are instant.

This three-stage pipeline, from training to analysis to visualization, allows me to spend more time interpreting the analysis than worrying about whether inconsistencies have crept in. When I encounter surprising or unexpected results in the visualizations, they are more likely to be real results. And those surprises give me more leads to follow into deeper analysis of how models learn.
